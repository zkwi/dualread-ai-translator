# ADR-0001：采用结构感知的翻译布局适配层

## 状态

Accepted

## 背景

网页正文可能位于块布局、Flex、Grid、列表、裁剪预览或 Web Component Slot 中。仅依赖译文节点的通用 `display:block;width:100%` 无法保证安全排版，站点专用 CSS 又会持续增加维护成本。同时，纯视口点采样追求速度，但会遗漏采样行之间的可见标题。

## 决策

在现有内容脚本中加入轻量结构感知层：

- 候选发现采用“快速采样 + 有界补扫”。
- 每个候选归一化为包含文本、安全锚点和布局模式的翻译单元。
- 译文根据 Block、Flex、Grid、裁剪和 Slot 上下文选择插入策略。
- 保留少量必要的平台语义识别，例如 Reddit Slot；不建立站点 class 配置表。

## 影响

### 正面

- 同一机制覆盖 Reddit、CNN 和其他同类卡片。
- 保持首段快速流式反馈，同时提升视口翻译完整性。
- 布局策略可以用本地夹具稳定回归。

### 负面

- 首次插入需要读取少量计算样式。
- Flex 容器需要在含译文时允许换行，卡片高度会增加。

### 中性

- 继续保留 Reddit Slot 等真正与平台 DOM 语义相关的适配。
- 所有布局标记使用扩展内部数据属性，并随译文清理。

## 备选方案

- 站点补丁：脆弱且持续增长，拒绝。
- 全局强制块样式：无法解决裁剪和不换行 Flex，拒绝。
- Shadow DOM 译文容器：能隔离样式但不能自动解决宿主页面的布局位置，复杂度过高，拒绝。
