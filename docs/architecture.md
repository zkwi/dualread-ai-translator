# 架构说明

DualRead 是一个无构建步骤的 Chrome Manifest V3 扩展。运行时由原生 JavaScript、Chrome Extension API 和 OpenAI-compatible Chat Completions API 组成，没有项目自有服务器。

## 文件职责

```text
shared.js       共享默认值、规范化、缓存 key、语言和布局纯函数
background.js   Chrome 生命周期、标签页控制、API 请求、SSE 解析和缓存
content.js      页面候选发现、逐段队列、译文插入和结构感知布局
popup.*         当前页面的开始/停止、显示方式和状态摘要
options.*       精简的连接/语言/阅读设置，以及折叠的高级覆盖与维护入口
_locales/       四种界面语言文案
test/           共享/后台单测、浏览器 fixture、扩展 smoke 和真实页面样本
```

项目刻意保持少文件、无打包器。只有职责出现可独立复用或测试的真实需求时才继续拆分。

## 正文翻译主链路

1. content script 在当前视口及附近发现可读文本块，并按正文/视口优先级排序。
2. 每个段落独立进入并发受限队列，默认同时运行 2 个请求。
3. content script 通过 `llm-translation-stream` Port 发送单段任务。
4. background 先查段落缓存；未命中则请求纯文本 SSE。
5. 每个 `delta` 立即更新页面中的“翻译中”占位，完成后保存缓存。
6. 端点明确不支持流式时，该 API/模型在当前 service worker 生命周期内降级为单段非流式请求。

`maxConcurrentBatches` 是历史存储键，当前语义为“同时翻译段落数”。`batchSize` 和 `maxCharsPerBatch` 仅为旧存储及兼容代码保留，不参与正文调度。

## 页面候选与布局

候选发现采用“快速视口采样 + 有界补扫”，不默认遍历整页。通用规则会跳过导航、广告、表单、弹窗、媒体错误层和低价值元数据。

译文插入前会识别 Block、Flex、Grid、列表、裁剪预览和 Web Component Slot。适配优先基于 DOM 结构和计算样式；只有无法用通用结构表达的平台语义才保留少量站点规则。

## Thinking 控制能力

自动模式不根据服务商、URL 或模型名称猜测参数。

- 点击“测试 API”时，background 依次尝试已有的通用 Thinking 控制形状。
- 只有 400/422 且明确指出 Thinking 字段不支持时才继续下一个候选。
- 鉴权、限流、模型、网络、超时和服务端错误立即停止。
- 首个成功策略与规范化的 API/模型检测键保存到本地存储。
- API 地址或模型变化后检测键不匹配，运行请求不发送额外 Thinking 参数，直到重新测试。
- 手动策略直接使用用户选择，不进行候选探测。

OpenAI-compatible 协议无法判断服务商是否静默忽略未知字段，因此自动结果表示“连接测试成功采用的参数”，不是对模型内部推理状态的证明。

## 缓存与预算

- 缓存按段落保存，key 包含 API 地址、模型、语言、提示词和原文。
- 缓存 TTL、最大条目数、每页请求数和字符数都有上限。
- 低层预算与缓存上限使用共享默认值；设置页不再暴露这些调优项，也不会覆盖旧存储中的自定义值。
- 停止/清除或切换运行时会取消旧请求，并忽略旧 run ID 的返回。

## 兼容边界

`translate_batch`、批量 JSON 提示词和修复解析器只兼容旧消息及回归测试。网页正文从 0.7.0 起不再调用它们。删除兼容层前应先确认没有已注入的旧 content script 或外部调用仍依赖该消息。
